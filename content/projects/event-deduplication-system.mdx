## The Problem

Openlane runs one of the largest automotive remarketing platforms in North America. Every vehicle that moves through the marketplace — sold at auction, offered for purchase, listed for inspection — goes through a workflow system that tracks its state from intake to completion.

By the time I inherited ownership of the inspection workflow service, it was handling 100K+ vehicles a day and showing the usual signs of a system that had grown faster than its architecture: processing was synchronous in places that needed to be async, state transitions had edge cases that surfaced only under load, and the deployment process was manual enough that rolling back a bad deploy was a stressful half-hour event rather than a five-minute one.

The mandate was to make the system reliable enough to run unattended — high throughput, fault-tolerant, and observable enough that problems were caught before customers noticed them.

## Architecture

The service is a .NET Core application built around a few key ideas:

**Event-driven async processing.** The most impactful change was pulling synchronous processing out of the critical path. Inspection steps that could be deferred — third-party data enrichment, notification dispatch, downstream system sync — were moved onto Apache Pulsar topics with dedicated consumer groups. This decoupled processing latency from API response time and gave us independent scaling and retry semantics for each step.

**Feature-flagged rollout via Terraform.** Production deployments now follow a canary pattern managed through Terraform-driven configuration. New logic ships behind a flag, gets routed to a configurable percentage of traffic, and promotes automatically when error rates and latency stay within bounds — or rolls back if they don't. On-call engineers no longer need to be present for every deploy.

**Redis caching for VIN decode.** VIN decoding was on the hot path for every vehicle intake, and we were paying vendor API costs per call. Adding a Redis caching layer with appropriate TTLs cut vendor API usage significantly and took VIN decode latency from a meaningful portion of intake time to essentially zero for cache-hit vehicles.

**Honeycomb observability.** The previous monitoring setup was alert-heavy and low-signal. I rebuilt the observability layer around Honeycomb, instrumenting the workflow funnel with structured events that let us trace a vehicle's lifecycle across services. Funnel dashboards show where vehicles drop out of the workflow. Alert thresholds are set on derived metrics — error rate by workflow stage, P95 per step — not just raw counts.

## Key Decisions

**Apache Pulsar over Kafka.** The decision was driven by existing infrastructure at Openlane — Pulsar was already the standard messaging layer. That said, Pulsar's multi-tenancy model and built-in tiered storage have been genuinely useful: we can namespace topics by workflow type and retain historical event streams without managing separate infrastructure.

**Canary deploys are non-negotiable at this scale.** At 100K+ vehicles/day, a bad deploy that affects 1% of traffic is still 1,000 vehicles per day. The Terraform-managed canary pattern was the right call. It added some upfront complexity to the deployment pipeline, but it's paid for itself many times over in avoided incidents.

**Instrument everything, alert on what matters.** Early in the project the alerting was noisy — too many alerts firing on raw counts that didn't distinguish between a real problem and a temporary upstream blip. Rebuilding around derived metrics (error rate, not error count; P95 per stage, not global P95) cut alert fatigue significantly and made the alerts that did fire more actionable.

## Results

- **100K+ vehicles/day** processed reliably in steady state with no SLA breaches during my ownership period
- **35% reduction in inspection processing latency** after moving synchronous steps to Pulsar-backed async processing
- **98% SLA maintained** across on-call rotations since the canary rollout strategy was adopted
- **70% improvement in VIN decode response time** and meaningful reduction in third-party vendor costs via Redis caching
- **45% reduction in MTTD** after rebuilding observability around Honeycomb funnel metrics and structured events

## What I'd Do Differently

The Pulsar consumer group topology grew organically, which meant some topics ended up with more partitions than they needed and others had too few for the throughput we were pushing through them. I'd design the topic partitioning strategy upfront based on projected load rather than adjusting it reactively.

On the observability side: I'd instrument the service from day one with the exact same Honeycomb event schema we ended up with, rather than retrofitting it. Retrofit instrumentation always leaves gaps — steps that weren't worth adding later, fields that required code changes to surface — and those gaps are exactly where the next incident hides.
